<objective>
Add the ADDITIVE per-KPI endpoint-pairing field to `swimlane.SectionBinding` that Phase-2's composer
needs to compute `compute_delta(start, close)` from two INDEPENDENTLY content-addressed endpoints
(COMP-02), and to keep the KPI strip kind-agnostic (COMP-01). Per the 02-PATTERNS.md "Endpoint-pairing
finding": the as-built `SectionBinding`/`KpiItem` does NOT expose which two Claims are a given KPI's
period endpoints — they are buried in the flat `claims` list with no back-reference, text-matching is
ambiguous (duplicate values), and endpoint counts are not recorded. Deriving delta from current output
is therefore unsafe. This plan closes that gap additively.

Purpose: give the composer an explicit, unambiguous pairing so a delta is only ever derived from two
traced endpoints — never guessed, never text-matched.
Output: one new optional `default_factory` field on `SectionBinding` populated by REFERENCE (not
re-minting) inside `_bind_kpis`, plus a dedicated proof test. This is the phase's early wave —
`compose.py` (Wave 2) depends on it.

This is an ADDITIVE-ONLY change under two HARD conditions (02-CONTEXT.md "Module shape"): every
existing Phase-1 test stays byte-untouched AND green; the abstraction guard still passes.
</objective>
