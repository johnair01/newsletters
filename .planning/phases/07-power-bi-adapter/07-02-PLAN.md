<objective>
Build the stdlib PBIR/report-JSON reader (`_pbir.py`) — PURE functions that turn a PBIR
`report.json`/`page.json`/`visual.json` object into (a) ordered `(object_path_prefix, verbatim_value)`
text units (page display names, visual titles, text-box runs, field references) and (b) a list of
TYPED row-cap/aggregation detections (the L3 taxonomy — the success-criterion-2 crux). The detector
finds the signals that make a clipped/aggregated export LOOK complete — TopN filters, restricting
filters, summarized/aggregated field bindings, DirectQuery/row-limit — so Wave 2 can route each to a
precise `unextracted[]` reason and force `Coverage.complete=False` (CONTEXT decision 4 / L3).

This plan resolves research Assumption A1 / Open-Q1 (exact PBIR `visual.json` filter key paths): it
encodes the detector against the documented PowerBI filter model (`filterType: TopN`, `operator`,
`itemCount`, aggregation function on a field projection) and the `microsoft/json-schemas`
visualContainer shape, and the detection is keyed leniently so a key-name variant degrades to "a
restricting filter is present" rather than a miss. Wave-3 pins the exact reason strings by driving the
LIVE adapter against the golden `visual.json`.

Purpose: put the genuinely-new row-cap detection behind a pure, unit-testable boundary (parallel to
Plan 01's TMDL parser, disjoint files) so Wave 2 composes both. Output: `src/newsletters/adapters/
_pbir.py` + `tests/test_pbir_reader.py`. ZERO new dependency (stdlib `json` only).
</objective>
