<objective>
Reorganise the dependency boundary so the deterministic spine installs and runs with ZERO AI
dependencies, and police that boundary with two complementary local gates. After this plan:
(1) `pyproject.toml` core `dependencies` are non-AI only (`pydantic>=2`, `typer[all]`, `sqlmodel`),
`pydantic-ai` lives in a new `[ai]` extra, and `langsmith`/`langchain[anthropic]`/`langgraph` are
gone entirely; (2) an import-linter `forbidden` contract statically bars `newsletters` core from
importing any AI package; (3) a runtime test catches what the import-linter structurally cannot —
pydantic-plugin auto-activation — by enumerating `entry_points(group="pydantic")` and `sys.modules`
after a fresh `import newsletters`.

Purpose: Satisfy PKG-01 (bare install = working pipeline, zero AI), PKG-02 (AI behind `[ai]`,
lazy-only), and PKG-04 (import-linter contract enforced). The CI workflow that makes these gates
a STANDING invariant on every push — including the canonical bare no-extras install gate PKG-03 —
is Plan 02-02, which depends on the contract + test this plan creates.

Output: A reorganised `pyproject.toml`; a new `.importlinter` config with a forbidden AI->core
contract; a new `tests/test_ai_optional.py` carrying the plugin-aware runtime isolation test, the
import-linter contract assertion, and a bare-pipeline smoke check.

WHY THE TWO GATES ARE BOTH NEEDED (load-bearing — from `.planning/notes/ai-optional-pydantic-plugin-leak.md`):
the import-linter (PKG-04) catches STATIC `import` edges from core to AI packages, but a pydantic
plugin (e.g. logfire's `logfire-plugin = logfire.integrations.pydantic:plugin`) activates with NO
import edge — pydantic auto-loads every installed plugin on first import. So the import-linter is
necessary-but-not-sufficient; the runtime `entry_points(group="pydantic")` + `sys.modules` assertion
is the complement. The TRUE source of truth (a bare no-extras install where the AI packages are
simply absent) is the CI job in Plan 02-02 (PKG-03). This is infra: keep it tight.
</objective>
