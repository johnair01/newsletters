<objective>
Turn the AI-optional boundary from a local property into a STANDING CI invariant. After this plan,
a GitHub Actions workflow runs on every push/PR with two jobs:

(1) `bare-install` (PKG-03 — the canonical source of truth): `pip install .` with NO extras into a
clean environment, then run the full deterministic pipeline (capture -> review -> render) and assert
no AI module is reachable from core. On a bare install the AI packages are simply absent, so the
pydantic-plugin auto-activation path (the import-linter's blind spot, per the leak-note) cannot fire
— this job is the real defense.

(2) `import-linter` (PKG-04): install `.[dev]`, run `lint-imports`, fail on any static `core -> AI`
import edge.

Purpose: PKG-03 and PKG-04 are not one-time tasks — STATE.md records them as "standing CI invariants
verified every subsequent phase." This workflow is what makes "the spine runs with zero AI" stay true
as Phases 3-14 add code. The contract + plugin-aware test it runs were built in Plan 02-01; this plan
wires them into the per-push gate. This is infra: one workflow file, two tight jobs.

Output: `.github/workflows/ci.yml` (no CI workflow exists today — verified live).

WHY BARE-INSTALL IS THE SOURCE OF TRUTH (load-bearing — leak-note:30-42): the import-linter catches
static import edges but MISSES pydantic-plugin activation (no import edge). The bare no-extras install
catches BOTH naturally: with AI packages uninstalled, there is no AI import to make AND no AI plugin
for pydantic to auto-activate. The import-linter job is the fast complementary static check; the
bare-install job is the definitive runtime proof.
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| later-phase code changes -> the AI-optional invariant | Every future commit could introduce a core->AI import edge or pull an AI package into the bare install; CI is the gate that catches it before merge. |
| CI runner's installed package set -> the "bare install" claim | The bare-install job's value depends on it installing NO AI extras; a stray `[ai]`/`[dev]` in that job would silently void the gate. |

> This phase ships **no runtime/network/auth/PII surface** — it is CI infrastructure. The genuine
> surface is the invariant being silently voided (a mis-scoped install) or an AI dependency slipping
> in unnoticed. We do not invent threats beyond these.

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-05 | Tampering | a future commit adds a core->AI import edge | mitigate | The `import-linter` job runs `lint-imports` on every push and fails on any core->AI edge (PKG-04 standing invariant). |
| T-02-06 | Tampering / Elevation | a future commit makes an AI package required (pulled into the bare install) | mitigate | The `bare-install` job installs no AI extra, asserts each AI module is NOT importable, AND runs the full pipeline — a required-AI regression fails the build (PKG-03). |
| T-02-07 | Information disclosure | AI/telemetry via pydantic-plugin activation slips back in | mitigate | On the bare install AI packages are absent, so no `pydantic`-group AI plugin can auto-activate; `test_no_ai_pydantic_plugin_active` runs here for the right reason (leak-note source-of-truth). |
